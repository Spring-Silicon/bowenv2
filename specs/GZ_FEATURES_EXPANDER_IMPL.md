# Expander Edge Emission Implementation Spec (Model Work Order A)

Status: implementation work order

Purpose: implement GZ_MODEL.md's FeatureSchema amendment — expander edges
emitted by the Rust extractor as a typed edge family, with the expander
config in `FeatureSchemaConfig` and its hash. After this work order the
wire carries everything the model trunk needs; the Python model work order
(GZ_MODEL_TORCH_IMPL.md) builds on it.

CONTEXT: GZ_TRAINING_DATA_IMPL.md is already implemented. This work order
therefore carries two ripples beyond gz-features: the replay store's
persisted schema-config mirror gains the new fields (replay SCHEMA_VERSION
bumps to 4; existing stores are dev-only and are regenerated, not
migrated), and the sample protocol's SHELLO_ACK is extended to ship the
full serialized FeatureSchemaConfig (the implemented ack predates that
amendment and ships only the hash — the Python trainer needs the config
to build its model).

Authority: `GZ_MODEL.md` (the amendment definition), `GZ_FEATURES.md`
(the crate contract being amended). Contract wins; report conflicts.

## Stage 1: Schema Fields And Hash

`FeatureSchemaConfig` gains:

```rust
pub expander_degree: u8,     // 0 disables emission entirely
pub expander_seed: u64,
```

Rules:

```text
Both fields hash into FeatureSchemaHash (length-delimited, after the
existing fields; every existing hash changes — that is the point of
landing this before any store exists).
Validation: when expander_degree > 0, max_edges must be >= the caller's
engine edge budget plus expander_degree * max_nodes. The schema cannot
know the engine budget, so the validation enforces only
max_edges >= expander_degree * max_nodes + 1; extractors own the full
budget (stage 2).
Edge type convention, documented on the config: when expander_degree > 0,
the expander edge type is edge_type_count - 1 (the last type), and
edge_type_count must include it. Engine edge types occupy 0..count-2.
FeatureRow::validate and the collator need no changes: expander edges
are ordinary typed edges within existing bounds.
```

Tests: hash changes when either new field changes; hash stable across
runs; validation rejects an undersized max_edges when degree > 0.

## Stage 2: Replay Store Ripple

`StoredFeatureSchemaConfig` in gz-replay gains `expander_degree: u8` and
`expander_seed: u64` (with the From conversions both ways). The persisted
meta encoding changes, so replay `SCHEMA_VERSION` bumps 3 -> 4; a v3 store
fails to open with SchemaMismatch (existing mechanism and test pattern).
Existing stores are development artifacts; regenerate, never migrate.
`ensure_feature_schema` equality naturally covers the new fields.

Tests: v4 roundtrip including the expander fields; the version gate; an
ensure mismatch on expander fields alone is rejected.

## Stage 3: Emission In The Whittle Extractor

`WhittleFeatureExtractor::new(engine)` derives its schema with:

```text
expander_degree = 5, expander_seed = 0   (constructor gains a config
                                          parameter with these defaults;
                                          degree 0 must remain expressible
                                          for tests)
edge_type_count = 3 (arg0, arg1, expander)
max_edges = 2 * capacity + expander_degree * capacity
```

Emission, per GZ_MODEL.md:

```text
expander edges for a graph with n real nodes = d permutations of 0..n,
each generated by seeded Fisher-Yates over a splitmix64 stream seeded by
a domain-prefixed mix of (expander_seed, permutation_index, n);
one edge (i, perm[i], EXPANDER_TYPE) per node per permutation, skipping
self-pairs (perm[i] == i); ONE direction only (the model mirrors).
Emission depends only on (expander_seed, n): cache the edge template per
node count inside the extractor (separate from the per-graph state
cache) and append it after the graph's arg edges during extract.
n <= 1 emits nothing. degree 0 emits nothing and leaves edge_type_count
at the engine's count (2 for Whittle).
```

Tests (gz-engine-whittle):

```text
same node count -> identical expander edges across extractor instances
and across calls (determinism + cache consistency)
different node counts -> independent templates; different seed ->
different edges
per-permutation self-pair skip: edge count == d * n - (number of fixed
points), all edges within bounds, all typed EXPANDER
degree 0 extractor emits arg edges only and reports edge_type_count 2
extract() still satisfies FeatureRow::validate under the enlarged
max_edges budget
```

## Stage 4: SHELLO_ACK Ships The Schema Config

Extend `handle_hello` in `crates/gz-cli/src/serve.rs`: after
`produced_rows`, append the serialized `FeatureSchemaConfig` exactly per
GZ_TRAINING_DATA_IMPL.md's field order (name_len u16 + utf8 name,
node_vocab_size u16, node_attr_dim u16, edge_type_count u8,
action_kind_vocab_size u32, max_nodes u32, max_edges u32, max_actions u32,
max_subjects u32, expander_degree u8, expander_seed u64). The
serialization helper belongs in gz-features (one codec; the Python trainer
and the checkpoint manifest reuse the same byte layout per
GZ_MODEL_TORCH_IMPL.md stage 1).

Update GZ_EVAL_PROTOCOL.md's Sample Protocol chapter to match, and update
the Rust test client + serve tests: the ack parse asserts the decoded
config equals the store's stored config, field for field.
